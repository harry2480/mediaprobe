# WASMメモリ最適化およびRAWデータダウンサンプリング方針

## 1. 目的（解決したい問題）

高画素のRAWファイル（特にDNGなど）を読み込んだ際、WASM（Rust）内で数十MB〜数百MBの巨大なメモリを一括で確保しようとするため、WASMのメモリ上限に達し `RuntimeError: unreachable`（Out Of Memoryエラー）が発生します。
この問題を解決しつつ、RAWデータの「リニアな階調」や「ダイナミックレンジの広さ（露出調整などの編集耐性）」を失わずにブラウザ上でプレビュー可能にする方針を策定します。

## 2. 解決の基本方針（サブサンプリング）

画像の全体像（画角）を保持したまま、画像を構成する画素（ピクセル）を均等に間引く（サブサンプリングする）ことで、出力される画像解像度を下げます。

- **「表示する部分」は維持**: クロップ（切り抜き）は行わず、画像全体の全体像を表示します。
- **「RAWの情報」は維持**: RAWデータの16bit整数や浮動小数点（Float）のリニアデータとしての性質は維持します。これにより、縮小された画像であっても、後段（React/Canvas側）での露出変更やガンマ補正などは、元のフル解像度データと全く同じように計算・適用できます。

## 3. 具体的な実装アプローチ

### Rust側の処理 (`wasm-processor/src/lib.rs`)

1. **最大解像度の設定**:
   プレビューとして十分な解像度（例: 長辺を最大 1920px 〜 2048px 程度）を設定します。
2. **リサイズ倍率とステップ（間引く間隔）の計算**:
   元の画像サイズと最大解像度を比較し、どれくらい間引けばよいか（何ピクセルごとにデータをサンプリングするか）を計算します。
   - 例: 元の幅が 6000px で目標が 2000px の場合、3ピクセルごとに1ピクセルを抽出する（ステップ幅 = 3）。
3. **間引き抽出による画像データの構築**:
   フル解像度のRGBA配列（`width * height * 4`）を確保するのではなく、**縮小後のサイズ（`new_width * new_height * 4`）** でメモリを確保します。配列の上限が劇的に減るため、WASMのメモリ枯渇を防げます。
   - `$y` と `$x` の2重ループでピクセルを抽出する際、ステップ幅を掛けて元データのインデックスを参照します。
   - Floatデータ、整数データの両バリアントに対してこの間引き処理を適用します。

### フロントエンド側の処理（必要に応じて）

1. Rustから返却されるデータは縮小済みのImageDataとなるため、JavaScript側の `ArrayBuffer`（または `Uint8Array`）にコピーされる際のメモリ負荷と処理時間も劇的に削減されます。
2. `<canvas>` 要素の見た目のサイズはCSSなどでコントロールされているため、フロントエンド（Reactコンポーネント）のロジックに大きな変更は生じません。受け取ったデータをそのままCanvasのコンテキストに `putImageData` します。

## 4. 期待される効果

- **メモリ枯渇（OOM）の完全な回避**: 確保するメモリ領域のサイズを1/4〜1/16程度に固定できるため、本番環境でのクラッシュを防げます。
- **パフォーマンスの劇的な向上**: メモリアクセス、配列のコピー処理、およびUIスレッドへのデータ転送にかかる負荷が大幅に低下し、描画速度が格段に向上します。
- **表示品質の維持**: 画像全体がプレビューでき、かつRAWのダイナミックレンジ情報が保持されるため、ユーザから見た機能性（露出スライダーによる白飛び・黒つぶれの確認など）は損なわれません。
